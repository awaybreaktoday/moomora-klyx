# Klyx open questions

Architectural and product questions to think through before writing too much code. These are deliberately unresolved - the answers shape the codebase.

## Data layer

**Q1: One informer factory per cluster, or one shared client with per-cluster watches?**

Informer factories are heavy (one cache per resource type), but they're how client-go is designed to be used. Spinning up six factories for six clusters in the fleet view means ~150MB of memory for caches. Alternative is a custom watch multiplexer, but that re-implements informers badly. Lean toward one factory per cluster, with lazy initialisation - only spin up watches for resource types currently being viewed.

**Q2: How do we handle cluster connection failures gracefully?**

When a cluster is unreachable (VPN dropped, credential expired, network blip), the fleet card should show a degraded state without blocking other clusters. The data layer needs per-cluster error isolation - one bad kubeconfig entry doesn't break the app. How does the UI signal "this card is stale, last successful refresh was 5 minutes ago"?

**Q3: Cross-cluster queries - aggregate at the data layer or the view layer?**

When the user searches "show all pods named share-api across the fleet", does the data layer expose a single `SearchPods(ctx, query)` that fans out to all clusters, or does the view layer make N calls and merge? Lean toward the data layer - it knows about cluster connection state and can handle partial failures cleanly.

## Capability detection

**Q4: How aggressive is detection?**

The minimum is CRD-presence detection (Flux CRDs installed = GitOps view enabled). But "installed" doesn't mean "actually working". A Flux installation with the controllers crashloop-backing-off is technically present but useless. Do we surface "Flux is installed but unhealthy" as a distinct state, or just lump it under "GitOps unavailable"?

**Q5: What about partial capabilities?**

Flux installed but no Notification controller. Gateway API installed but no EnvoyProxy. Cilium installed without Hubble. These are real and common. The capability model needs to be finer-grained than "tool present yes/no".

## GitOps

**Q6: Argo ApplicationSets - how to render?**

An ApplicationSet generates many Applications. Do we show the ApplicationSet as a parent node with generated Applications as children, or flatten them into the same list with a "generated by X" annotation? The former is structurally honest; the latter is easier to scan.

**Q7: Flux suspend semantics.**

Suspending a Kustomization stops reconciliation but doesn't roll back. Suspending a HelmRelease stops upgrades. These are different operations with the same UI verb. Does the suspend button have a confirmation dialog explaining the actual effect for that resource type?

**Q8: Reconcile-on-click.**

Triggering an immediate reconcile via the `reconcile.fluxcd.io/requestedAt` annotation is dangerous in production (could trigger a bad deploy faster than normal). Does prd-tagged cluster require an extra confirmation step?

## Network topology

**Q9: How do we render policies that affect a route?**

A given HTTPRoute can have a BackendTrafficPolicy attached, a ClientTrafficPolicy attached to its Gateway, AND a CiliumNetworkPolicy that restricts pod-to-pod traffic on its backend. Three policy systems, attached at different layers. Show all three on the route view, or show them only on the relevant node (Gateway/Route/Service)?

**Q10: Non-Envoy ingress controllers.**

A homelab k3s probably has Traefik, not Envoy Gateway. An EKS lab might have AWS Load Balancer Controller. Klyx should gracefully render whatever's installed - but each controller has different CRDs and conventions. How much per-controller code is acceptable vs falling back to a generic "Ingress" view?

## Multi-cluster

**Q11: ClusterMesh detection.**

Detecting Cilium ClusterMesh requires more than checking for the Cilium CRD - need to look at the actual mesh config and the connected peers. What's the minimum signal that says "this cluster is meshed with that one"?

**Q12: Cluster grouping in the fleet view.**

When you have 6 clusters, no grouping is needed. When you have 30, you want to group by environment, region, or provider. The user should be able to choose the grouping dimension. What's the data model for this?

## Observability

**Q13: PromQL query templates per resource.**

For a Pod, you want CPU and memory. For an HTTPRoute, you want p50/p99/rps. For a Flux Kustomization, you want reconciliation duration. These are different PromQL templates per resource type. Where do they live - hardcoded, config file, or user-editable?

**Q14: Multi-source Prometheus.**

Each cluster typically has its own Prometheus. Klyx needs to know which Prom to query for which cluster. Does the user configure this per-cluster, or does Klyx detect the Prom service in each cluster automatically?

## UX

**Q15: Command palette scope.**

Does `⌘K` search the current cluster, or all clusters? If all, results need to be grouped/filtered by cluster. If current, switching clusters loses search context.

**Q16: Pinned resources / favourites.**

If you're investigating a problem, you want to keep certain resources visible across navigation. Tabs? Pinned items in the sidebar? A separate "workspace" concept?

**Q17: Stale data indication.**

Informer caches can drift from the API server during connection issues. The UI needs to indicate "this data was last refreshed N seconds ago" without being noisy. What's the threshold for "this is fine" vs "this is concerning"?

## Distribution and ops

**Q18: Telemetry.**

Does Klyx phone home for crash reports, usage analytics, or version checks? Default should be off, opt-in only. If on, what's collected?

**Q19: Update channel.**

Stable vs beta. How does the user opt in?

**Q20: `klyx serve` headless mode auth.**

The headless server exposes the same UI as a web app. Auth via Entra ID/Google/GitHub OIDC. RBAC enforcement - does it inherit from the cluster's RBAC (impersonation), or does the server have its own RBAC layer? Impersonation is cleaner architecturally but harder to configure.
